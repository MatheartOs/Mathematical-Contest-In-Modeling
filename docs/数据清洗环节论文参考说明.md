# 数据清洗环节论文参考说明

> 本文档用于支撑数模赛论文中“数据预处理与文件画像构建”部分的撰写。内容对应当前项目实现 `src/mcm_b/cleaning.py` 与全量清洗结果 `outputs/b_problem/cleaning_ocr_full/`。若后续代码或参数变化，应同步更新本文档与 `docs/PROJECT_MAIN.md`。

## 1. 清洗目标

B 题的数据来源具有明显的多源异构特征：既包含普通文本、Word、Excel、PDF，也包含大量图片文件、图片侧车说明文本以及结构化表格数据。直接将原文件输入建模算法会产生三个问题：

1. 不同格式的文本载体无法统一比较。
2. 图片与扫描件缺少可检索文本，容易造成历史主题体系缺失。
3. 表格、正文、附件、日期、金额、机构等业务信息混杂，难以支撑后续分类与复核优先级建模。

因此，本项目的清洗目标不是简单提取纯文本，而是构建统一的文件画像：

```text
D_i = {T_i, L_i, S_i, B_i, Q_i}
```

其中：

| 符号 | 含义 | 当前实现中的对应字段 |
| --- | --- | --- |
| `T_i` | 文件文本内容，包括标题、正文、表格语义文本、OCR 文本 | `title`, `full_text`, `clean_text` |
| `L_i` | 块级版面信息，包括页码、块类型、bbox、阅读顺序 | `document_blocks.jsonl` |
| `S_i` | 文件结构特征，包括页数、段落数、表格数、图片数、标题数等 | `page_count`, `paragraph_count`, `table_count`, `image_count`, `heading_count` |
| `B_i` | 业务属性，包括金额、日期、截止、机构、联系人和业务关键词 | `date_list`, `deadline`, `amount_list`, `organization_list`, `business_keywords` |
| `Q_i` | 解析质量，包括解析方式、OCR 置信度、缺失率、人工复核标记 | `parse_method`, `ocr_confidence`, `parse_quality`, `need_manual_check` |

该设计使后续问题一可以基于清洗后的统一文本和业务属性建立主题体系，问题二可以将新数据投影到同一特征空间，问题三可以利用质量和风险字段决定人工复核顺序。

## 2. 输入数据范围

当前清洗脚本扫描项目根目录下的官方数据目录：

```text
B题数据集/
```

清洗入口由 `src/mcm_b/paths.py` 定义，主要包括：

| 数据来源 | 说明 | 统一编号前缀 |
| --- | --- | --- |
| 数据集 1 | 历史真实文件，问题一的主题体系主要依据 | `D1_` |
| 数据集 2 | 后续流入文件，问题二分类对象之一 | `D2_` |
| 数据集 3 | 表格行级记录，按行展开为虚拟文档 | `D3_` |
| 数据集 4 | 资源约束数据，供问题三使用 | 当前仅扫描记录，不作为清洗文本主体 |

每个被扫描对象会先进入 `file_manifest.csv`，获得统一文件编号，例如 `D1_0001`、`D2_0001`、`D3_0001`。这样可以在论文中说明：原始文件名、文件路径和建模内部编号之间存在可追踪映射，便于复核与回溯。

## 3. 总体流程

清洗流程可概括为七个阶段：

```text
原始文件扫描
  -> 文件类型识别
  -> 格式化解析 / OCR 识别
  -> 文本标准化
  -> 块级结构构建
  -> 业务属性抽取
  -> 质量评估与人工复核标记
  -> 标准化产物输出
```

在论文中可描述为：先对所有文件建立文件清单，再依据扩展名和文件内容进入不同解析分支；所有分支的输出最终被统一映射为文件级索引表和块级结构表，从而为后续建模提供一致的数据接口。

## 4. 文件扫描与编号

扫描阶段生成 `processed/file_manifest.csv`。该表用于记录文件的基础元数据，字段如下：

```text
file_id, original_id, dataset_id, file_name, file_path, file_ext,
file_size_kb, modified_time, is_readable, initial_type, parse_status
```

字段含义：

| 字段 | 含义 |
| --- | --- |
| `file_id` | 清洗系统内部统一编号 |
| `original_id` | 原文件名去除扩展名后的编号或名称 |
| `dataset_id` | 所属数据集，如 `dataset1`、`dataset2`、`dataset3` |
| `file_name` | 原始文件名 |
| `file_path` | 原始文件路径 |
| `file_ext` | 文件扩展名 |
| `file_size_kb` | 文件大小，单位 KB |
| `modified_time` | 文件修改时间 |
| `is_readable` | 文件是否可读 |
| `initial_type` | 初步判定类型，如 text、word、excel、pdf、image |
| `parse_status` | 解析状态，初始为 pending |

编号规则采用“数据集前缀 + 四位序号”，既能保留来源，又能避免不同数据集之间文件名冲突。

## 5. 类型分流策略

清洗模块根据扩展名和内容特征将文件分为以下类别：

| 类别 | 扩展名或来源 | 解析策略 | `parse_method` 示例 |
| --- | --- | --- | --- |
| 文本文档 | `.txt` | 多编码尝试读取，识别普通文本或图片侧车说明 | `txt_parse:utf-8-sig`, `image_sidecar_txt:utf-8-sig` |
| Word 文档 | `.docx` | 解压 OOXML，读取段落、表格、标题和图片信息 | `docx_parse` |
| Excel 表格 | `.xlsx`, `.xls`, `.csv` | 读取工作表，将表头和单元格转为语义文本 | `excel_parse` |
| PDF 文档 | `.pdf` | 先尝试文本抽取，短文本或扫描件标记为待 OCR | `text_pdf`, `scanned_pdf_ocr_pending` |
| 图片文件 | `.jpg`, `.jpeg`, `.png`, `.bmp`, `.tif`, `.tiff` | 使用 PaddleOCR 识别文字 | `image_paddleocr` |
| 数据集 3 行记录 | 表格行 | 每一行转为一个虚拟文本文件 | `dataset3_row_parse` |
| 超大文件 | 超过大小阈值 | 只保留元数据，防止单文件拖垮流程 | `metadata_only` |

当前全量清洗的分流结果为：

| 解析方式 | 数量 |
| --- | ---: |
| `dataset3_row_parse` | 3518 |
| `image_paddleocr` | 1990 |
| `image_sidecar_txt:utf-8-sig` | 1127 |
| `docx_parse` | 617 |
| `txt_parse:utf-8-sig` | 351 |
| `excel_parse` | 176 |
| `text_pdf` | 76 |
| `scanned_pdf_ocr_pending` | 54 |
| `scanned_pdf_text_low` | 5 |
| `metadata_only` | 2 |

需要特别说明的是，`image_sidecar_txt` 指“图片名称、图片编号、下载 URL”等图片元数据说明文件。它被保留用于追踪，但不作为主题文本主体，否则会把大量“图片名称/下载链接/编号”噪声带入聚类。

## 6. 各类文件的清洗细节

### 6.1 TXT 文本

TXT 文件存在编码不统一的问题。当前按以下顺序尝试解码：

```text
utf-8-sig -> utf-8 -> gb18030 -> gbk -> utf-16 -> latin-1
```

若文本中出现以下结构：

```text
图片名称:
图片编号:
下载URL:
```

则将其标记为图片侧车文本 `image_sidecar_txt:*`。该类文件在索引中保留，但在后续主题建模中应谨慎使用或剔除。

### 6.2 DOCX 文档

DOCX 本质上是 OOXML 压缩包。当前解析策略包括：

1. 解压文档结构。
2. 提取正文段落。
3. 提取表格文本。
4. 统计标题、段落、表格、图片等结构特征。
5. 检测附件、签章、落款等业务结构线索。

DOCX 输出的段落和表格会进入 `document_blocks.jsonl`，形成块级表示。

### 6.3 Excel 表格

Excel 文件的清洗重点不是保留原始单元格格式，而是将表格转为可参与文本建模的语义表示。当前流程为：

1. 遍历工作表。
2. 读取有效行列。
3. 将表头与单元格内容拼接为文本片段。
4. 统计表格数量。
5. 将表格块写入块级表。

该方法可以保留表格中常见的项目名称、金额、日期、单位、状态等业务字段。

### 6.4 PDF 文档

PDF 先进行文本抽取：

1. 若可抽取到足够文本，则标记为 `text_pdf`。
2. 若文本过短或疑似扫描件，则标记为 `scanned_pdf_ocr_pending` 或 `scanned_pdf_text_low`。

当前版本尚未实现“PDF 转图片后调用 PaddleOCR”。因此全量结果中仍有 54 个 `scanned_pdf_ocr_pending`，这是后续可优化点。

### 6.5 图片文件

图片文件是本题清洗的重点。当前全部图片均使用 PaddleOCR 识别，解析方式为 `image_paddleocr`。OCR 输出包括：

| 输出 | 写入位置 |
| --- | --- |
| OCR 拼接全文 | `document_index.csv` 的 `full_text`, `clean_text` |
| 单行识别文本 | `document_blocks.jsonl` |
| 单行置信度 | `document_blocks.jsonl` 的 `confidence` |
| bbox 坐标 | `document_blocks.jsonl` 的 `bbox` |
| 平均置信度 | `document_index.csv` 和 `ocr_log.csv` 的 `ocr_confidence` |

全量结果中，1990 个图片文件全部成功完成 OCR，平均置信度为 0.989121。

## 7. OCR 模型与运行环境

当前项目只使用 PaddleOCR，不使用其他 OCR 引擎。

### 7.1 OCR 框架

| 项目 | 当前配置 |
| --- | --- |
| OCR 框架 | PaddleOCR |
| PaddleOCR 版本 | `3.5.0` |
| PaddlePaddle GPU 版本 | `paddlepaddle-gpu==3.0.0.dev20250717` |
| 运行设备 | `gpu:0` |
| GPU | NVIDIA GeForce RTX 5090 D |
| Driver | `596.21` |
| Driver CUDA 显示 | `13.2` |
| Paddle Runtime | CUDA 12.9 相关运行库 |

Paddle CUDA 验证结果：

```text
paddle.__version__ == 3.0.0
paddle.device.is_compiled_with_cuda() == True
paddle.set_device("gpu:0") 成功
```

### 7.2 本地 OCR 模型

使用的 PP-OCRv5 server 模型位于 ASCII 路径：

```text
C:\mcm_paddleocr_models\det\PP-OCRv5_server_det_infer
C:\mcm_paddleocr_models\rec\PP-OCRv5_server_rec_infer
```

采用 ASCII 路径是因为早期测试发现 Paddle 底层在中文项目路径下读取本地模型时可能出现 JSON 解析异常。为保证竞赛期间跨终端稳定复现，模型统一放在 `C:\mcm_paddleocr_models`。

代码中模型目录优先级为：

1. 环境变量 `PADDLEOCR_DET_DIR` 与 `PADDLEOCR_REC_DIR`。
2. `C:\mcm_paddleocr_models\det\PP-OCRv5_server_det_infer` 与 `C:\mcm_paddleocr_models\rec\PP-OCRv5_server_rec_infer`。
3. 项目内 `models/paddleocr/`。

### 7.3 OCR 初始化参数

当前 PaddleOCR 初始化逻辑位于 `src/mcm_b/cleaning.py`：

```python
PaddleOCR(
    text_detection_model_dir=str(det_dir),
    text_recognition_model_dir=str(rec_dir),
    use_doc_orientation_classify=False,
    use_doc_unwarping=False,
    use_textline_orientation=False,
    device=device,
    enable_mkldnn=False,
)
```

参数说明：

| 参数 | 含义 | 当前取值原因 |
| --- | --- | --- |
| `text_detection_model_dir` | 文本检测模型目录 | 使用本地 PP-OCRv5 server det 模型 |
| `text_recognition_model_dir` | 文本识别模型目录 | 使用本地 PP-OCRv5 server rec 模型 |
| `use_doc_orientation_classify` | 是否启用文档方向分类 | 当前关闭，减少额外模型依赖 |
| `use_doc_unwarping` | 是否启用文档矫正 | 当前关闭，保持流程稳定 |
| `use_textline_orientation` | 是否启用文本行方向分类 | 当前关闭，避免额外模型下载 |
| `device` | 运行设备 | 默认 CUDA 可用时为 `gpu:0` |
| `enable_mkldnn` | 是否启用 CPU MKLDNN | 当前关闭；CPU 测试中 MKLDNN 曾触发兼容错误，GPU 下保留关闭不影响主流程 |

设备选择规则：

```text
若设置 PADDLEOCR_DEVICE，则使用该值；
否则若 Paddle 编译支持 CUDA，则使用 gpu:0；
否则回退到 cpu。
```

建议竞赛期间全量清洗显式设置：

```powershell
$env:PADDLEOCR_DEVICE='gpu:0'
```

## 8. 文本标准化

不同文件解析后会形成初始 `full_text`，随后统一清洗为 `clean_text`。主要标准化操作包括：

1. 全角字符转半角。
2. 合并多余空白符。
3. 去除明显无意义的控制字符。
4. 使用业务同义词进行归一化。
5. 截断过长文本，避免单文件过度支配 TF-IDF。

当前清洗脚本的文本保留上限为：

```text
--max-chars 30000
```

这意味着每个文件最多保留 30000 个字符进入索引表。该限制可避免极端长文档导致内存膨胀，同时保留足够上下文供主题识别。

## 10. 块级结构表

除了文件级索引，清洗还生成块级结构表：

```text
processed/document_blocks.jsonl
```

每一行是一个 JSON 对象，描述一个文本块。核心字段包括：

| 字段 | 含义 |
| --- | --- |
| `file_id` | 所属文件编号 |
| `block_id` | 块编号 |
| `page` | 页码，图片 OCR 默认第 1 页 |
| `block_type` | 块类型，如 paragraph、table、image_text |
| `text` | 块文本 |
| `bbox` | OCR 坐标框，非 OCR 文本可为空 |
| `source` | 来源，如 paddleocr、docx、excel |
| `confidence` | 置信度，OCR 块为识别置信度 |
| `reading_order` | 阅读顺序 |

对于图片 OCR，`block_type` 为 `image_text`，`source` 为 `paddleocr`。这使得论文可以说明：本项目不仅保留 OCR 拼接文本，还保留了 OCR 行级结果与空间位置，便于后续人工复核和可解释性分析。

## 11. 质量评估

每个文件都会生成质量字段：

| 字段 | 含义 |
| --- | --- |
| `parse_success` | 是否解析成功 |
| `parse_quality` | 综合解析质量分 |
| `missing_rate` | 缺失率估计 |
| `need_manual_check` | 是否需要人工复核 |
| `manual_check_reason` | 触发人工复核的原因 |
| `ocr_confidence` | OCR 平均置信度 |
| `ocr_page_count` | OCR 页数或图片数量 |

当前质量评分综合考虑：

1. 文本长度是否足够。
2. OCR 置信度是否可靠。
3. 是否保留结构信息。
4. 是否存在解析错误。
5. 文件格式是否完整读取。

可在论文中写为：

```text
Q_i = 0.30 q_text + 0.20 q_ocr + 0.20 q_layout + 0.20 q_complete + 0.10 q_format
```

其中 `q_text` 表示文本充足性，`q_ocr` 表示 OCR 质量，`q_layout` 表示结构完整性，`q_complete` 表示解析完整性，`q_format` 表示格式读取稳定性。当前代码将其汇总为 `parse_quality`。

人工复核触发规则包括：

| 触发原因 | 说明 |
| --- | --- |
| `parse_quality_below_0.5` | 解析质量较低 |
| `parse_quality_between_0.5_and_0.7` | 解析质量中等，需要抽检 |
| `text_too_short` | 文本过短，信息不足 |
| `ocr_required_or_pending` | OCR 文件或 OCR 待处理文件 |
| `PDF text extraction too short; OCR required.` | 扫描 PDF 尚待 OCR |
| `File exceeds max size budget` | 文件超过大小预算，仅保留元数据 |

当前全量清洗中 `manual_check_list.csv` 共 6781 行，主要原因如下：

| 人工复核原因 | 数量 |
| --- | ---: |
| `parse_quality_between_0.5_and_0.7` | 3471 |
| `ocr_required_or_pending` | 1541 |
| 图片侧车 TXT 需原图/OCR 内容 | 1127 |
| `parse_quality_between_0.5_and_0.7; ocr_required_or_pending` | 432 |

需要注意：人工复核标记并不等同于解析失败。大量 OCR 文件被标记复核，是因为 OCR 天然需要质量控制，而不是 OCR 未完成。

## 16. 与后续建模的衔接

清洗结果进入后续建模时，主要使用：

```text
outputs/b_problem/cleaning_ocr_full/processed/document_index.csv
```

建模文本通常由以下部分组合：

```text
model_text = title + clean_text + business_keywords
```

后续链路中：

1. 问题一使用数据集 1 中解析质量足够的历史文件，基于 `clean_text` 和 `business_keywords` 建立主题体系。
2. 问题二将数据集 2 和数据集 3 的清洗文本投影到同一特征空间，判断其主题归属。
3. 问题三结合 `has_deadline`、`has_urgent`、`parse_quality`、`ocr_confidence`、分类置信度等字段计算人工复核优先级。

因此，清洗环节不仅是预处理步骤，也是问题三风险控制变量的重要来源。

## 17. 可写入论文的方法表述

可在论文中采用如下表述：

```text
针对题目给出的多源异构办公文件，本文首先构建统一文件画像。对每个文件记录基础元数据，并根据文件扩展名和内容特征分流至文本、Word、Excel、PDF、图片 OCR 和表格行解析模块。对图片类文件，采用 PaddleOCR 的 PP-OCRv5 server 检测模型和识别模型，在 GPU 环境下提取行级文字、置信度和坐标框。所有解析结果统一写入文件级索引表 document_index.csv 和块级结构表 document_blocks.jsonl。

在文本标准化阶段，本文对不同来源文本进行编码统一、空白归并、业务同义词归一和长度控制；同时抽取日期、截止时间、金额、机构、联系方式及八类业务关键词。最后，从文本充分性、OCR 置信度、版面结构、解析完整性和格式稳定性五个方面计算解析质量分，并生成需要人工复核的文件列表。
```

也可写为数学化流程：

```text
RawFile_i -> Parser(type_i) -> Blocks_i
Blocks_i -> Text_i, Layout_i, Structure_i
Text_i -> Business_i
{Text_i, Layout_i, Structure_i, Business_i} -> Quality_i
```

最终得到：

```text
D_i = {T_i, L_i, S_i, B_i, Q_i}
```

其中 `D_i` 是第 `i` 个文件的统一画像。

## 18. 当前局限与后续改进

当前清洗已经完成全部图片 OCR，但仍有以下局限：

1. 扫描 PDF 尚未转图片 OCR，当前 54 个扫描 PDF 被标记为 `scanned_pdf_ocr_pending`。
2. OCR 对超长图片会触发自动缩放到 `max_side_limit=4000`，可能轻微影响极小文字识别。
3. 人工复核规则偏保守，导致 `manual_check_list.csv` 数量较高；论文中可解释为质量控制而非错误率。
4. Excel 表格目前主要转为语义文本，未完全保留复杂表格拓扑关系。
5. 图片侧车 TXT 保留但不作为主题正文，若需要更细粒度溯源，可进一步建立侧车 TXT 与原图的一一映射表。

后续若时间允许，优先增强 PDF 转图片 OCR；其次可对低置信度 OCR 图片进行抽样人工复核，并将复核结果反向校正质量阈值。
