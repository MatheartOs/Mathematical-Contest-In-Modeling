# 多源异构文件数据清洗流程说明文档

> 适用项目：2026 年第十一届数维杯大学生数学建模挑战赛（春季赛）B 题  
> 主题：智能办公场景下多源异构文件识别与治理优化  
> 文档用途：用于后续数据清洗操作指导、团队任务分工、代码开发交接、论文方法部分撰写参考。  
> 版本：v1.0  
> 建议维护人：数据预处理负责人  

---

## 0. 本阶段目标

本阶段的核心目标不是直接分类，而是先将 Word、PDF、Excel、TXT、JPG/PNG、扫描件等多源异构文件转化为统一、可计算、可追踪的标准数据格式。

最终应形成两类核心产物：

1. **文件级索引表**：`document_index.csv`  
   每个文件一行，记录文件基本信息、文本信息、结构信息、业务信息、质量信息。

2. **内容块级结构表**：`document_blocks.jsonl`  
   每个文件拆分为若干文本块、表格块、图片块，保留页码、块类型、文本内容、位置坐标、来源方式等信息。

后续三个问题的关系如下：

```text
数据清洗
  ↓
文件画像构建
  ↓
问题一：历史文件分类与主题体系建立
  ↓
问题二：新流入文件归属判断与迁移评价
  ↓
问题三：风险分级、人工复核决策与资源约束优化
```

## 1. 清洗总原则

### 1.1 不只提取纯文本

本题文件属于典型的多源异构办公文件，部分文件可能具有明显版面结构、表格结构、图片信息或扫描特征。

因此，清洗目标不应只是获得一段 `full_text`，还应尽可能保留：

- 标题；
- 段落；
- 表格；
- 图片；
- 页码；
- OCR 置信度；
- 日期；
- 金额；
- 部门/机构；
- 文件格式；
- 文件结构；
- 解析质量；
- 是否需要人工检查。

### 1.2 文本、结构、业务、质量四类信息同时保留

每个文件建议最终抽象为：

```text
D_i = {T_i, L_i, S_i, B_i, Q_i}
```

其中：

| 符号 | 含义 |
|---|---|
| `T_i` | 文本内容，包括标题、正文、表格转写文本、OCR 文本 |
| `L_i` | 布局信息，包括页码、文本块坐标、阅读顺序 |
| `S_i` | 结构信息，包括段落数、页数、表格数、图片数 |
| `B_i` | 业务属性，包括资金、项目、会议、通知、合同等特征 |
| `Q_i` | 清洗质量，包括 OCR 置信度、解析完整度、缺失情况 |

### 1.3 不过度清洗

不要删除对建模有用的信息，例如：

- 日期；
- 金额；
- 编号；
- 标题序号；
- 单位名称；
- 部门名称；
- 文件落款；
- “截止”“紧急”“报送”“审批”“经费”等业务关键词。

这些内容在问题二和问题三中非常重要。

## 3. 阶段一：建立文件清单

### 3.1 目标

在正式解析文件前，先扫描所有附件文件，建立文件总清单。

### 3.2 需要记录的字段

生成 `file_manifest.csv`：

| 字段名 | 含义 |
|---|---|
| `file_id` | 文件唯一编号 |
| `dataset_id` | 所属数据集，如 dataset1、dataset2、dataset3 |
| `file_name` | 原始文件名 |
| `file_path` | 文件路径 |
| `file_ext` | 文件后缀 |
| `file_size_kb` | 文件大小 |
| `modified_time` | 文件修改时间，如可获取 |
| `is_readable` | 文件是否可读取 |
| `initial_type` | 初步文件类型 |
| `parse_status` | 待解析、已解析、失败 |

### 3.3 文件编号规则

建议统一编号：

| 数据集 | 编号方式 |
|---|---|
| 数据集 1 | `D1_001`, `D1_002`, ... |
| 数据集 2 | `D2_001`, `D2_002`, ... |
| 数据集 3 | `D3_001`, `D3_002`, ... |
| 数据集 4 | `D4_001`, `D4_002`, ... |

如果文件本身已有编号，应保留原编号，同时增加标准编号。

## 4. 阶段二：文件类型识别与分流

### 4.1 总体分流逻辑

```text
读取文件后缀
  ↓
判断文件是否可打开
  ↓
按类型进入对应解析器
  ↓
若解析失败，记录错误并进入人工检查列表
```

### 4.2 文件类型及处理策略

| 文件类型 | 常见后缀 | 处理策略 |
|---|---|---|
| Word | `.docx`, `.doc` | 提取标题、段落、表格、页眉页脚 |
| Excel | `.xlsx`, `.xls`, `.csv` | 提取工作表、表头、单元格、金额和日期 |
| PDF | `.pdf` | 先判断文字型、扫描型、混合型，再分流 |
| TXT | `.txt` | 编码识别、文本清洗、段落切分 |
| 图片 | `.jpg`, `.jpeg`, `.png` | 图像预处理 + OCR |
| 其他 | 其他格式 | 尝试通用解析，失败则人工标记 |

## 5. 阶段三：PDF 文件专项处理

PDF 是最复杂的数据来源，建议单独处理。

### 5.1 PDF 类型判断

对每个 PDF 先尝试直接提取文本。

设：

```text
R_text = 直接提取出的文字数 / PDF 页数
```

判断规则建议如下：

| 条件 | PDF 类型 | 后续处理 |
|---|---|---|
| `R_text >= 100` | 文字型 PDF | 直接文本提取为主 |
| `20 <= R_text < 100` | 混合型 PDF | 直接提取 + 低文字页 OCR |
| `R_text < 20` | 扫描型 PDF | 转图片后 OCR |
| 提取失败 | 异常 PDF | 记录日志并人工检查 |

阈值可以根据实际数据调整。

### 5.2 文字型 PDF 处理

提取内容：

- 页码；
- 正文文本；
- 标题候选；
- 段落；
- 表格；
- 页眉页脚；
- 图片数量；
- 是否含印章、签名、附件说明。

### 5.3 扫描型 PDF 处理

处理流程：

```text
PDF 每页转图片
  ↓
图像预处理
  ↓
OCR 识别
  ↓
文本块排序
  ↓
记录 OCR 置信度
  ↓
合并为页面文本
```

图像预处理建议包括：

- 灰度化；
- 去噪；
- 倾斜校正；
- 对比度增强；
- 裁剪黑边；
- 分辨率检查。

### 5.4 混合型 PDF 处理

混合型 PDF 采用双通道策略：

1. 先直接提取文本；
2. 对文本极少的页面进行 OCR；
3. 对直接文本和 OCR 文本去重合并；
4. 标记该文件为 `parse_method = mixed_pdf`.

### 5.5 PDF 输出字段

每个 PDF 页面建议保存：

```json
{
  "file_id": "D1_001",
  "page": 1,
  "text": "本页提取文本",
  "tables": [],
  "images": [],
  "ocr_used": true,
  "ocr_confidence": 0.87,
  "source": "pdf_ocr"
}
```

## 6. 阶段四：Word 文件处理

### 6.1 提取内容

Word 文件应提取：

- 文件名；
- 标题候选；
- 正文段落；
- 一级标题；
- 二级标题；
- 表格；
- 页眉；
- 页脚；
- 落款；
- 日期；
- 附件说明。

### 6.2 标题识别规则

标题可由以下来源综合判断：

1. 文件名；
2. 第一段非空文本；
3. 字号较大或居中加粗文本；
4. 包含“通知”“方案”“报告”“总结”“申请”“合同”等关键词的短文本。

建议保留多个标题候选：

```json
{
  "title_from_filename": "关于开展项目申报工作的通知",
  "title_from_first_paragraph": "关于开展项目申报工作的通知",
  "final_title": "关于开展项目申报工作的通知",
  "title_confidence": 0.95
}
```

### 6.3 表格处理

Word 中表格不能直接丢弃，应转为两种形式：

1. 原始二维表；
2. 表格语义文本。

示例：

```text
该表包含字段：姓名、部门、项目名称、预算金额、截止时间。
表中出现关键词：预算、经费、负责人、提交。
```

## 7. 阶段五：Excel 文件处理

### 7.1 提取内容

Excel 文件应提取：

- 工作表名称；
- 表头；
- 行数；
- 列数；
- 单元格内容；
- 合并单元格；
- 金额字段；
- 日期字段；
- 统计字段；
- 空值比例。

### 7.2 表格结构保留

Excel 文件建议输出：

```json
{
  "file_id": "D1_010",
  "sheet_name": "经费预算表",
  "n_rows": 35,
  "n_cols": 8,
  "headers": ["项目名称", "负责人", "预算金额", "申请单位"],
  "table_text": "本表包含项目名称、负责人、预算金额、申请单位等字段。"
}
```

### 7.3 Excel 文本化策略

为便于后续分类，需要将表格结构转化为可建模文本：

```text
工作表名称：经费预算表。
表头字段：项目名称、负责人、预算金额、申请单位、审批状态。
主要关键词：经费、预算、金额、项目、负责人、审批。
```

### 7.4 不建议做法

不要直接把整个 Excel 所有单元格拼成一长串文本。

原因：

- 会破坏行列结构；
- 金额和项目之间的对应关系会丢失；
- 表头信息的重要性会被削弱；
- 后续解释性变差。

## 8. 阶段六：TXT 文件处理

### 8.1 编码识别

TXT 文件可能存在编码问题，建议自动识别：

- UTF-8；
- GBK；
- GB2312；
- UTF-16。

如识别失败，应记录到 `error_log.txt`。

### 8.2 文本清洗内容

处理步骤：

1. 统一换行符；
2. 删除异常乱码；
3. 删除多余空行；
4. 保留段落边界；
5. 提取标题；
6. 提取日期、金额、机构等信息。

### 8.3 注意事项

不要简单删除所有标点。中文办公文件中的冒号、括号、书名号、顿号、序号等通常具有结构意义。

## 9. 阶段七：图片文件处理

### 9.1 处理流程

```text
读取图片
  ↓
图像质量检查
  ↓
图像预处理
  ↓
OCR 识别
  ↓
文本行排序
  ↓
输出文本、坐标、置信度
```

### 9.2 图像质量检查

建议记录：

| 字段 | 含义 |
|---|---|
| `image_width` | 图片宽度 |
| `image_height` | 图片高度 |
| `dpi` | 分辨率，如可获取 |
| `is_blurry` | 是否模糊 |
| `is_rotated` | 是否旋转 |
| `quality_flag` | 高、中、低 |

### 9.3 OCR 输出要求

OCR 结果应保留：

| 字段 | 含义 |
|---|---|
| `text` | 识别文本 |
| `bbox` | 文本框坐标 |
| `confidence` | OCR 置信度 |
| `line_order` | 阅读顺序 |
| `source` | OCR 工具名称或处理方式 |

## 10. 阶段八：统一文本标准化

所有文件提取出文本后，需要进行统一标准化。

### 10.1 基础标准化

| 操作 | 说明 |
|---|---|
| 全角转半角 | 统一数字、字母、标点 |
| 繁体转简体 | 如数据中存在繁体 |
| 删除乱码 | 删除不可见字符、异常编码字符 |
| 统一空白 | 合并多余空格、空行 |
| 合并断行 | 修复 PDF/OCR 导致的异常换行 |
| 去除重复页眉页脚 | 删除每页重复出现的页眉页脚 |
| 去除页码噪声 | 删除“第 x 页”“Page x”等无效信息 |

### 10.2 日期标准化

将不同日期格式统一为 `YYYY-MM-DD`。

示例：

| 原始形式 | 标准形式 |
|---|---|
| `2026 年 5 月 8 日` | `2026-05-08` |
| `2026/5/8` | `2026-05-08` |
| `2026.05.08` | `2026-05-08` |
| `5月8日前` | 结合年份推断为 `2026-05-08` |

### 10.3 金额标准化

将金额统一为“元”。

示例：

| 原始形式 | 标准形式 |
|---|---|
| `3万元` | `30000元` |
| `1.5 万元` | `15000元` |
| `12,500.00元` | `12500元` |
| `伍万元` | `50000元`，如能识别中文数字 |

### 10.4 同义词归一

建议建立业务同义词词典。

| 原词 | 归一词 |
|---|---|
| 经费、预算、拨款、补贴、报销 | 资金 |
| 会议安排、会议通知、参会 | 会议 |
| 申报、立项、验收、结题 | 项目 |
| 协议、合同、责任书 | 合同 |
| 通知、公告、公示 | 通知 |

## 11. 阶段九：关键信息抽取

### 11.1 日期抽取

抽取所有日期，并重点识别截止时间。

日期关键词上下文：

- 截止；
- 之前；
- 前；
- 报送；
- 提交；
- 完成；
- 上报；
- 逾期；
- 限期。

示例：

```text
请于2026年5月20日前提交材料
```

抽取结果：

```json
{
  "deadline": "2026-05-20",
  "deadline_context": "提交材料",
  "has_deadline": 1
}
```

金额关键词上下文：

- 经费；
- 预算；
- 金额；
- 拨款；
- 报销；
- 补贴；
- 合计；
- 支出；
- 收入。

示例：

本项目经费预算为3.5万元

### 11.3 机构与部门抽取

- 学院；
- 部；
- 处；
- 办公室；
- 中心；
- 委员会；
- 公司；
- 单位；
- 小组；

### 11.4 人名、地点、联系方式抽取

可以使用正则表达式和 NER 工具抽取：

- 地点；
- 电话；

### 11.5 关键词抽取

## 13. 阶段十一：结构特征提取

### 13.1 文件结构特征

建议提取以下字段：

| 字段 | 含义 |
|---|---|
| `page_count` | 页数 |
| `paragraph_count` | 段落数 |
| `table_count` | 表格数量 |
| `image_count` | 图片数量 |
| `heading_count` | 标题数量 |
| `heading_depth` | 标题层级深度 |
| `avg_paragraph_length` | 平均段落长度 |
| `max_paragraph_length` | 最长段落长度 |
| `has_signature` | 是否存在落款 |
| `has_attachment` | 是否提到附件 |
| `has_stamp_or_seal` | 是否可能存在印章或盖章 |

### 13.2 结构特征用途

| 结构现象 | 可能含义 |
|---|---|
| 表格较多，金额较多 | 资金预算、统计表、清单类 |
| 标题短、落款明显、日期明显 | 通知公告类 |
| 条款编号多，甲乙方明显 | 合同协议类 |
| 段落长，标题层级多 | 工作报告、总结、方案类 |
| 含时间、地点、参会人员 | 会议纪要类 |
| 含申报、负责人、预算、截止时间 | 项目申报类 |

## 14. 阶段十二：清洗质量评分

### 14.1 为什么要做质量评分

清洗质量会影响后续分类准确性。

如果某个文件：

- OCR 置信度低；
- 提取文本过短；
- 表格解析失败；
- 日期金额缺失；
- 文件损坏；
- 扫描图片模糊；

那么该文件后续分类置信度应降低，并可能进入人工复核队列。

### 14.2 质量评分模型

建议定义：

```text
Q_i = 0.30 Q_text + 0.20 Q_ocr + 0.20 Q_layout + 0.20 Q_complete + 0.10 Q_format
```

其中：

| 指标 | 含义 |
|---|---|
| `Q_text` | 文本长度和有效字符比例 |
| `Q_ocr` | OCR 置信度 |
| `Q_layout` | 版面结构保留程度 |
| `Q_complete` | 关键信息完整度 |
| `Q_format` | 文件是否成功解析 |

### 14.3 子指标建议

#### 1. 文本质量 `Q_text`

```text
Q_text = min(text_length / expected_length, 1)
```

如果文本过短，则质量较低。

#### 2. OCR 质量 `Q_ocr`

| 情况 | 分数 |
|---|---|
| 未使用 OCR 且直接解析成功 | 1.0 |
| OCR 平均置信度 ≥ 0.90 | 0.9 |
| OCR 平均置信度 0.75-0.90 | 0.75 |
| OCR 平均置信度 0.60-0.75 | 0.6 |
| OCR 平均置信度 < 0.60 | 0.4 |

#### 3. 布局质量 `Q_layout`

| 情况 | 分数 |
|---|---|
| 段落、表格、页码均保留 | 1.0 |
| 保留段落和页码，表格部分缺失 | 0.8 |
| 仅保留纯文本 | 0.6 |
| 文本顺序混乱 | 0.4 |

#### 4. 完整度 `Q_complete`

可由标题、正文、日期、业务关键词等字段缺失情况计算。

```text
Q_complete = 1 - missing_key_field_count / total_key_field_count
```

#### 5. 格式解析质量 `Q_format`

| 情况 | 分数 |
|---|---|
| 成功解析 | 1.0 |
| 部分解析 | 0.7 |
| 解析失败但 OCR 成功 | 0.6 |
| 解析失败 | 0.0 |

### 14.4 人工检查标记

建议规则：

| 条件 | 标记 |
|---|---|
| `Q_i < 0.5` | 必须人工检查 |
| `0.5 <= Q_i < 0.7` | 建议人工抽查 |
| `Q_i >= 0.7` | 可进入自动建模流程 |

输出字段：

```text
parse_quality = Q_i
need_manual_check = 0/1
manual_check_reason = "OCR置信度低；文本过短；表格解析失败"
```

## 24. 清洗阶段常见问题与处理建议

### 24.1 OCR 文本顺序混乱

处理方法：

- 按文本框纵坐标排序；
- 同一行内按横坐标排序；
- 对多栏文档进行栏识别；
- 保留原始 OCR 块，避免无法回溯。

### 24.2 PDF 页眉页脚重复

处理方法：

- 统计每页重复出现的短文本；
- 若出现频率超过 70%，判为页眉页脚候选；
- 从正文中删除，但在元信息中保留。

### 24.3 Excel 合并单元格导致表头缺失

处理方法：

- 向下或向右填充合并单元格值；
- 保留原始表格；
- 表头缺失时使用前几行共同判断。

### 24.4 标题识别错误

处理方法：

- 文件名、首段、字号、关键词共同投票；
- 保留多个标题候选；
- 设置 `title_confidence`。

### 24.5 文本极短

处理方法：

- 判断是否为图片或扫描件；
- 尝试 OCR；
- 若仍极短，标记为 `short_text = 1`；
- 进入人工检查列表。

### 24.6 金额识别误差

处理方法：

- 同时支持阿拉伯数字和中文大写数字；
- 识别“万元”“元”“千元”等单位；
- 保留金额上下文，避免把编号误识别为金额。
